## Task 1: Fix Free Order Registration (CRITICAL)

**The Bug:** When someone registers for a free event on the public event page, `POST /api/orders` creates tickets and orderItems but never inserts a `guests` record. Free registrants are completely invisible in the Guest List and Check-in app. The tickets also have `guestId: null`, so QR barcode check-in silently "succeeds" without writing to any DB record.

**File:** `src/app/api/orders/route.ts`

**Step 1: Locate the free order ticket creation loop (line ~164)**

The current loop (line 166–196) creates tickets without a preceding guest insert. The fix is to insert the guest first, then insert the ticket referencing `guest.id`.

**Step 2: Replace the ticket creation loop**

Find this block:
```typescript
for (const item of cartItems) {
  const ticketType = validTicketTypes.find(tt => tt.id === item.ticketTypeId);
  for (let i = 0; i < item.quantity; i++) {
    const barcode = `TKT-${nanoid(12).toUpperCase()}`;
    const [newTicket] = await db
      .insert(tickets)
      .values({
        companyId: company[0].id,
        eventId: event[0].id,
        ticketTypeId: item.ticketTypeId,
        orderId: order[0].id,
        barcode,
        attendeeName,
        attendeeEmail,
        status: "valid",
      })
      .returning();
```

Replace with:
```typescript
for (const item of cartItems) {
  const ticketType = validTicketTypes.find(tt => tt.id === item.ticketTypeId);
  for (let i = 0; i < item.quantity; i++) {
    const barcode = `TKT-${crypto.randomBytes(6).toString("hex").toUpperCase()}`;

    // Create guest record first so it appears in Guest List & Check-in app
    const [firstName, ...lastNameParts] = attendeeName.trim().split(/\s+/).filter(Boolean);
    const [newGuest] = await db
      .insert(guests)
      .values({
        companyId: company[0].id,
        eventId: event[0].id,
        firstName: firstName || "Guest",
        lastName: lastNameParts.join(" "),
        email: attendeeEmail,
        status: "confirmed",
        guestType: ticketType?.name ?? item.name,
        source: "registration",
      })
      .returning();

    const [newTicket] = await db
      .insert(tickets)
      .values({
        companyId: company[0].id,
        eventId: event[0].id,
        ticketTypeId: item.ticketTypeId,
        orderId: order[0].id,
        guestId: newGuest.id,       // ← links ticket to guest
        barcode,
        attendeeName,
        attendeeEmail,
        status: "valid",
      })
      .returning();
```

## Task 3: Fix Bulk Selection Using Wrong Indices (HIGH)

**The Bug:** `guests/page.tsx` lines 535 and 545 map row selection to guests like:
```typescript
Object.keys(rowSelection).map(idx => data?.guests?.[parseInt(idx)]?.id)
```
TanStack Table's `rowSelection` uses row indices **within the rendered table**, not the original data array. If the DataTable applies client-side sorting or the data changes between renders, `parseInt(idx)` maps to the wrong guest. The correct approach is to get IDs from the table's selected rows directly.

**File:** `src/app/(dashboard)/dashboard/events/[eventId]/guests/page.tsx`

**Step 1: Check how DataTable exposes row data**

The table instance's `getSelectedRowModel().rows` gives actual row objects. However, since `rowSelection` is managed externally as `Record<string, boolean>`, the simplest safe fix is to **key row selection by the guest's UUID** (not index) in the DataTable.

**Step 2: Update `DataTable` to use row ID as selection key**

Open `src/components/shared/data-table.tsx`. Find where `useReactTable` is configured and add `getRowId`:

```typescript
const table = useReactTable({
  data,
  columns,
  // ...existing config...
  getRowId: (row) => (row as { id: string }).id,   // ← add this line
  state: {
    rowSelection,
    // ...rest
  },
});
```

**Step 3: Update the bulk action ID extraction in guests/page.tsx**

Replace both occurrences of the index-based mapping:
```typescript
// OLD (fragile index-based):
const selectedIds = Object.keys(rowSelection)
  .map(idx => (data?.guests?.[parseInt(idx)] as Guest)?.id)
  .filter(Boolean);

// NEW (key is already the guest UUID):
const selectedIds = Object.keys(rowSelection).filter(Boolean);
```

Apply to both the "Bulk Check In" handler (~line 535) and the "Undo Check In" handler (~line 545).

## Task 4: Update `tickets.checkedIn` on Guest Check-In (MEDIUM)

**The Bug:** The `guests.checkIn` mutation in `src/server/trpc/routers/guests.ts` updates `guests.status` and `guests.checkedInAt` but never touches the corresponding `tickets.checkedIn` and `tickets.checkedInAt` columns. The Tickets list in the dashboard will always show tickets as un-checked-in.

**File:** `src/server/trpc/routers/guests.ts`

**Step 1: Locate the checkIn mutation (around line 312)**

After the existing code that updates the guest status:
```typescript
const [updatedGuest] = await ctx.db
  .update(guests)
  .set({ status: "checked_in", checkedInAt: new Date(), updatedAt: new Date() })
  .where(eq(guests.id, input.guestId))
  .returning();
```

**Step 2: Add ticket sync after the guest update**

```typescript
// Also mark the linked ticket as checked-in
if (ticket) {
  await ctx.db
    .update(tickets)
    .set({ checkedIn: true, checkedInAt: new Date(), updatedAt: new Date() })
    .where(eq(tickets.id, ticket.id));
}
```

**Step 3: Locate the undoCheckIn mutation (around line 373)**

After the guest revert:
```typescript
const [updatedGuest] = await ctx.db
  .update(guests)
  .set({ status: "confirmed", checkedInAt: null, updatedAt: new Date() })
  ...
```

**Step 4: Add ticket sync for undo**

First fetch the ticket, then revert it:
```typescript
const [linkedTicket] = await ctx.db
  .select()
  .from(tickets)
  .where(
    and(
      eq(tickets.guestId, input.guestId),
      eq(tickets.eventId, input.eventId)
    )
  )
  .limit(1);

if (linkedTicket) {
  await ctx.db
    .update(tickets)
    .set({ checkedIn: false, checkedInAt: null, updatedAt: new Date() })
    .where(eq(tickets.id, linkedTicket.id));
}
```
